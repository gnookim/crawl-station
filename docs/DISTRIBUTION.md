# CrawlStation 분배 정책 및 크롤링 전략

## 핵심 원칙

1. **차단 회피** — 네이버에서 봇으로 탐지되지 않는 자연스러운 행동
2. **자원 효율** — 워커 간 균등 분배, 결과 반환 속도 유지
3. **수평 확장** — 워커당 할당량 제한, 워커 수를 늘려서 처리량 확보

---

## 작업 유형

### Type A. 키워드 상위 컨텐츠 심화 분석

키워드 검색 → 통합검색 + 영역탭(블로그/지식인/카페) → 상위 N개 컨텐츠 내부까지 진입하여 세부 분석.

| 항목 | 내용 |
|------|------|
| 목적 | 상위노출 컨텐츠 분석 → AI로 이길 수 있는 컨텐츠 생성 |
| 작업 범위 | 키워드 1개 → 통합검색 + 블로그탭 + 지식인탭 + 카페탭 각각 상위 N개 |
| 특징 | 한 키워드에 많은 페이지 방문 필요 (검색 + N개 상세페이지) |
| 빈도 | 요청 시 (온디맨드) |
| 분배 | 서브태스크 분할 — 영역별로 다른 워커에 분산 가능 |

```
예시: "강남 피부과" 심화 분석 요청
  워커 A → 통합검색 + 블로그탭 상위 10개 분석
  워커 B → 지식인탭 상위 10개 분석
  워커 C → 카페탭 상위 10개 분석
```

**최종 목표**: 분석 결과를 AI에 전달 → 기존 상위 컨텐츠를 이기는 컨텐츠 자동 생성

### Type B. 통합검색 영역 순위 분석

키워드 검색 → 통합검색에 표시되는 영역(섹션)들의 존재 여부 + 순서 파악.

| 항목 | 내용 |
|------|------|
| 목적 | 키워드별 검색 결과 구조 파악 (어떤 영역이 노출되는지) |
| 대상 영역 | 파워링크, 지식인, 블로그, 인기 카페글, 웹문서, 클립, 쇼핑 등 |
| 특징 | 통합검색 1페이지만 확인하면 됨 (상대적으로 가벼운 작업) |
| 빈도 | 요청 시 또는 정기 스케줄 |
| 분배 | 키워드 단위로 워커에 균등 분배 |

```
예시: "강남 피부과" 영역 분석 결과
  1위: 파워링크 (광고)
  2위: 플레이스
  3위: 블로그 (VIEW탭)
  4위: 지식인
  5위: 카페
  6위: 웹문서
```

### Type C. 일일 순위 체크 (대규모 반복 작업)

등록된 URL(우리 컨텐츠)이 통합검색 + 해당 영역 탭에서 몇 위에 있는지 매일 체크.

| 항목 | 내용 |
|------|------|
| 목적 | 우리 컨텐츠의 검색 순위 일일 트래킹 |
| 규모 | ~1만개 컨텐츠 + 매일 새로 등록되는 컨텐츠 |
| 체크 방식 | 키워드 검색 → 통합검색에서 URL 존재 확인 + 해당 탭에서 순위 |
| 빈도 | 매일 1회 (24시간 내 완료) |
| 분배 | **가장 중요한 분배 대상** — 워커 수를 늘려서 해결 |

```
예시: 1만개 URL, 워커 10대
  워커당 1,000개 → 각 워커가 하루 중 시간대를 나눠서 처리
  워커 A: 00시~02시 (500개), 12시~14시 (500개)
  워커 B: 02시~04시 (500개), 14시~16시 (500개)
  ...
```

**핵심 주의사항**:
- 연속 실행 금지 — 1,000개를 쉬지 않고 돌리면 탐지됨
- 시간대 분산 — 하루 중 여러 타임슬롯에 나눠서 실행
- 워커 수 확보 — 워커 3대로는 부족, 10대 이상 권장

### Type D. 지식인 경로 분석

키워드 검색 시 지식인 컨텐츠의 유입 경로 판별.

| 항목 | 내용 |
|------|------|
| 목적 | 지식인 상위노출 컨텐츠가 자체등록인지 닥톡(외부 서비스) 경로인지 파악 |
| 판별 기준 | 컨텐츠 페이지 내부의 출처/경로 정보 분석 |
| 빈도 | Type A 심화 분석의 일부로 실행 |
| 분배 | Type A의 지식인 서브태스크에 포함 |

## 분배 정책

### 현재 구현 상태

| 방식 | 설명 | 사용 시점 |
|------|------|----------|
| 선착순 (polling) | 워커가 5초마다 DB 폴링, 먼저 가져간 워커가 실행 | POST /api/crawl |
| 라운드 로빈 | dispatch API가 워커에 균등 배정 | POST /api/dispatch |

### 목표 분배 정책

#### 1. 워커별 일일 할당량

```
worker_config 테이블에 추가:
  daily_quota: 500        # 하루 최대 작업 수
  daily_used: 0           # 오늘 처리한 작업 수 (자정 리셋)
  quota_reset_at: "..."   # 마지막 리셋 시각
```

- 할당량 소진 시 해당 워커에 작업 배정 안 함
- 다른 워커에 자동 재배정
- Station 대시보드에서 할당량 조절 가능

#### 2. 서브태스크 분할 (Type A 심화 분석)

하나의 키워드 분석 요청을 여러 서브태스크로 분할:

```
원본 요청: { keyword: "강남 피부과", type: "deep_analysis" }

분할 후:
  subtask_1: { keyword: "강남 피부과", scope: "blog_tab",     parent_id: "원본" }  → 워커 A
  subtask_2: { keyword: "강남 피부과", scope: "kin_tab",      parent_id: "원본" }  → 워커 B
  subtask_3: { keyword: "강남 피부과", scope: "cafe_tab",     parent_id: "원본" }  → 워커 C
  subtask_4: { keyword: "강남 피부과", scope: "integrated",   parent_id: "원본" }  → 워커 A

모든 서브태스크 완료 → 결과 병합 → 원본 요청 completed
```

장점:
- 한 워커가 같은 키워드로 여러 탭을 연속 조회하지 않음 (패턴 분산)
- 병렬 처리로 결과 반환 속도 향상
- 한 워커 차단돼도 다른 서브태스크는 계속 진행

#### 3. 시간대 분산 스케줄러 (Type C 일일 순위)

```
daily_schedule:
  total_urls: 10,000
  workers: 10
  per_worker: 1,000
  time_slots_per_worker: 4    # 하루 4번에 나눠서
  per_slot: 250               # 한번에 250개씩
  slot_duration: ~2시간       # 250개 × (검색 + 딜레이) ≒ 2시간
  rest_between_slots: 4시간   # 슬롯 간 4시간 휴식

워커별 스케줄 예시:
  워커 A: 01시, 07시, 13시, 19시
  워커 B: 02시, 08시, 14시, 20시
  워커 C: 03시, 09시, 15시, 21시
  ...
```

#### 4. 작업 유형별 우선순위

```
Type A (심화 분석):   priority 10  — 사용자가 직접 요청, 즉시 처리
Type B (영역 분석):   priority 5   — 중간 우선순위
Type C (일일 순위):   priority 1   — 백그라운드, 하루 안에만 완료
Type D (경로 분석):   priority 10  — Type A에 포함, 동일 우선순위
```

높은 priority 요청이 들어오면 Type C 작업을 일시 중단하고 먼저 처리.

#### 5. 워커 쿨다운

```
crawl_config:
  batch_size: 30              # 연속 작업 수
  batch_rest_min: 180         # 배치 후 휴식 (초)
  batch_rest_max: 300
  task_delay_min: 15          # 작업 간 딜레이 (초)
  task_delay_max: 30
  daily_rest_hours: [3, 4, 5] # 이 시간대에는 쉼 (새벽 3~5시)
  warmup_sites: 3             # 작업 시작 전 워밍업 사이트 수
```

## 봇 탐지 회피 계층

### Layer 1: 네트워크 (이미 구현)
- IP 로테이션: 프록시, 테더링 자동 전환
- 배치 간 IP 변경

### Layer 2: 브라우저 핑거프린트 (이미 구현)
- User-Agent 랜덤 (Chrome/Safari/Edge/모바일)
- canvas/WebGL fingerprint 노이즈
- hardwareConcurrency, deviceMemory 랜덤화
- webdriver 속성 숨김

### Layer 3: 행동 패턴 (이미 구현)
- 세션 워밍업 (네이버 메인/뉴스/쇼핑 사전 방문)
- 타이핑 속도 변화, 오타
- 마우스 커서 이동, 스크롤
- 랜덤 결과 클릭 후 돌아오기
- 읽는 척 멈추기 (1~3초 pause)

### Layer 4: 분배 (구현 완료)
- 워커별 일일 할당량 제한 (daily_quota, atomic increment)
- 서브태스크 분할 (deep_analysis → 4개 영역별 분산)
- 시간대 분산 (Vercel Cron + slot_hours)
- 작업 간 자연스러운 간격 (15~30초 + 배치 휴식)
- 새벽 시간대 휴식 (KST 3~5시)

### Layer 5: 패턴 위장 (구현 완료)
- 목적 없는 검색 섞기 (25개 일반 키워드, 15% 확률)
- 검색 → 스크롤 → 30% 확률 결과 클릭 → 뒤로가기
- 세션 워밍업 (네이버 메인/뉴스/쇼핑 사전 방문)
